# Project Intake（项目接入分析）

## 核心原则

用户说"分析"时要的是**结论**,不是原始数据。agent 应该读完文件后沉默地综合,然后输出结构化卡片——而不是把 `cat README.md` 的结果直接甩给用户。

## 何时使用

- 用户说"分析当前项目"、"分析下项目"、"分析项目"。
- 用户说"分析一下README.md"、"读取README.md"并期望得到摘要。
- 用户进入一个新项目目录,第一次对话时说 hello 或简单问候。
- 用户说"这个项目是做什么的"、"项目概览"。

## 方法论

### 分析的五个维度

1. **身份**:项目是什么、解决什么问题、一句话概述。
2. **技术栈**:语言、框架、运行时、包管理器、部署目标。
3. **架构骨架**:目录结构、入口文件、核心模块划分。
4. **配置与约束**:环境变量、构建命令、测试命令、部署方式。
5. **活跃度**:最近 commit、贡献者、版本号、CHANGELOG 状态。

### 信息采集策略

按**成本递增**的顺序采集,每一步都可能已经足够产出卡片:

1. `ls -la` 项目根目录 → 识别文件类型和目录结构
2. `cat package.json` / `Cargo.toml` / `go.mod` / `pyproject.toml` → 技术栈和依赖
3. `cat README.md` → 项目自述（只提取关键信息,不要全文输出）
4. `find . -maxdepth 2 -type f | head -50` → 目录骨架
5. `git log --oneline -10` → 近期活跃度
6. `rg` 快速扫描入口文件和关键模块 → 架构理解

### 输出格式

始终输出以下结构化项目卡片,不要输出原始文件内容:

```markdown
## 项目卡片

**一句话概述**: <用一句话说清楚这个项目是什么、做什么>

### 技术栈
| 维度 | 值 |
|---|---|
| 语言 | <语言及版本> |
| 框架 | <框架> |
| 运行时 | <Node/Bun/Deno/...> |
| 包管理 | <npm/bun/pnpm/cargo/...> |
| 部署目标 | <Cloudflare Workers/Vercel/Docker/...> |

### 目录骨架
<用 tree 风格展示关键目录和文件,标注每个顶层目录的职责>

### 关键模块
- **<模块名>**: <一句话职责>（`<文件路径>`）
- ...

### 构建与运行
- 安装: `<命令>`
- 开发: `<命令>`
- 测试: `<命令>`
- 构建/部署: `<命令>`

### 近期活跃
- 最近 commit: `<日期> — <摘要>`
- 版本: `<版本号>`（如可获取）
- 总 commit 数 / 近 30 天 commit 数（如可获取）

### 已知约束 / 注意事项
<从 README、配置文件、或代码注释中提取的约束条件,如端口限制、API key 要求等>
```

## 操作步骤

1. **静默采集**:依次执行上述信息采集步骤,不要把原始输出展示给用户。
2. **综合分析**:把采集到的信息交叉验证,形成项目卡片。
3. **输出卡片**:按上述模板输出结构化项目卡片。
4. **主动补充**:如果发现 README 过时、配置缺失、或有明显问题,在卡片末尾的"已知约束"里注明。

## 配合的 agent

- `project-analyzer` agent:只读分析型 agent,负责执行信息采集和卡片生成。
