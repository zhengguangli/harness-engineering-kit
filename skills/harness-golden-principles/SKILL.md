# Golden Principles & Garbage Collection(黄金原则与垃圾回收)

## 为什么会有熵增

Agent 会忠实地复制仓库里已经存在的模式——**包括不太好的模式**。如果代码库里某处有一个不太优雅的 helper 函数被写了三次而不是被抽成共享工具,下一个 agent 大概率会照着抄第四次。高吞吐量放大了这一点:腐化速度和产出速度成正比。

技术债是一笔**利息很高的贷款**:持续小额还款,远比攒到无法忍受时才痛苦地集中处理划算。这正是这个技能要解决的——把"定期人工大扫除"换成"持续的小颗粒度自动清扫"。

## 黄金原则 vs 架构边界,两者不要混淆

| | `harness-architecture-boundaries` | `harness-golden-principles` |
|---|---|---|
| 性质 | 结构性不变量(依赖方向、数据边界) | 习惯/品味一致性(命名、复用、惯用法) |
| 强制时机 | 合并前阻塞(CI 红线) | 合并后周期性清扫 |
| 违反后果 | 必须修复才能合并 | 进入清扫队列,低风险情况下可自动合并修复 |
| 典型例子 | "Service 层不能直接 import Runtime 内部模块" | "优先用共享工具包而不是手写 helper""不要对数据做 YOLO 式动态取值,要走类型化 SDK 或显式校验边界" |

把这两者分开很重要:不要把品味偏好做成阻塞合并的硬规则,会拖慢吞吐量又不增加真正的安全边际;也不要把真正的结构性约束只当成"建议",那等于没有约束。

## 怎么提炼一条黄金原则

不要凭空发明规则。流程是:

1. **从真实信号里捕捉模式**:review 意见里反复出现的同一类反馈、重复出现的用户可见 bug、重构 PR 里反复修的同一类问题。
2. **把这条意见从"这次怎么改"提炼成"以后都应该怎么做"**的通用陈述。
3. **把它编码成机械规则**:能写成 lint 就写成 lint,不能写成 lint 的就先写进文档,作为下一步被巡检 agent 检索/比对的依据。
4. **一次性捕捉,持续生效**:人类的判断只需要给一次,之后每一行代码都会被这条规则检查,不需要每次 review 都重新讲一遍同样的话。

## 周期性清扫的运作方式

```
按固定节奏(例如每日/每周)运行一次清扫任务:
  1. 扫描代码库,对照已编码的黄金原则寻找偏差
  2. 给受影响的领域/分层更新质量评分(参考 QUALITY_SCORE.md)
  3. 对每个发现的偏差,生成一个独立、小颗粒度的修复 PR
     —— 范围限定到"一分钟内能审完"的大小,不要把多处不相关的修复
        塞进同一个 PR
  4. 风险评估:
     - 纯机械、无行为变化的修复(重命名、抽公共方法、
       替换为标准 SDK 调用)→ 可配置为自动合并
     - 涉及行为变化的修复 → 必须走人工或更高权限的评审
  5. 把本轮未处理/暂缓处理的发现记录进
     docs/exec-plans/tech-debt-tracker.md,而不是悄悄丢弃
```

清扫任务的目标不是"一次性清干净",而是**让偏差的存活时间趋近于零**——一个不一致的模式出现后,几天内就会被下一轮清扫发现并修复,而不是积累几个月后才被注意到。

## 操作步骤

1. 和用户一起盘点:最近的 review/bug 里有没有反复出现的同一类反馈?把它们列出来作为候选黄金原则。
2. 为每条候选原则判断:能写成 lint 吗?如果能,优先写成 lint;如果不能,先写进文档并明确"这是品味原则,靠周期性扫描而不是 CI 阻塞来强制"。
3. 设计清扫任务的运行节奏和范围限定规则(每个修复 PR 的最大改动量、哪些类型的修复可以自动合并)。
4. 把清扫任务接入定期触发(配合 `entropy-collector` agent),并明确它产出的 PR 要小到能被快速审完。
5. 持续观察:哪些黄金原则被频繁触发?说明那个模式还没有被从根本上修(比如该建一个共享工具包但还没建)——把这个信号本身记录进 `tech-debt-tracker.md`。

## 配合的 agent

- `entropy-collector` agent:执行上面描述的周期性清扫,产出小颗粒度修复 PR。
