# Observability & Browser Verification(可观测性与浏览器验证)

## 核心原则:把"信不信"换成"看不看"

人对 agent 生成代码天然有信任壁垒——模型是非确定性的、不真正"理解"代码、不知道你的隐含上下文。建立信任的方式不是说服,而是**让 agent 的每一个判断都基于它亲眼看到的真实信号**:浏览器里实际渲染出来的状态、日志里实际打出来的错误、指标里实际测到的延迟。这些是"反馈传感器"(feedback sensors)——没有它们,agent 只能靠读代码静态推断,推断和现实之间永远有一道无法消除的缝。

## 何时使用

- 需要复现 UI bug 或验证某条用户旅程是否正常工作。
- 需要确认性能/可靠性约束(如启动时间、P99 延迟)是否满足。
- 没有自动化手段确认改动是否真的修好了问题。

## 两类反馈传感器

### 1. 浏览器驱动验证

适用于:UI bug 复现、交互流程验证、视觉回归检查。

标准循环:

```
选定目标(要复现的 bug / 要验证的用户旅程)
  → 触发前拍快照(DOM 状态 / 截图)
  → 执行触发路径(点击、输入、导航)
  → 观察运行时事件(控制台报错、网络请求、状态变化)
  → 诊断问题
  → 实施修复(*)
  → 重启应用
  → 重新执行同一条路径
  → 触发后拍快照,和触发前对比
  → 循环直到状态符合预期
```

> (*) 注意:修复和重启由执行型 agent(如 verification-loop-runner)完成;qa-verifier 只负责到"诊断问题"和"拍摄对比证据",不自行修复。

如果项目里已经配置了浏览器自动化工具(例如某个 Playwright/Chrome DevTools 相关的 MCP 工具),优先使用它;没有配置时,提醒用户这是一个值得补上的环境能力缺口,而不是退回去"读代码猜测 UI 行为"。

**关键产出物**:修复前后的截图/录屏对比。这不只是给人看的证据,也是 agent 自己判断"这次真的修好了"的依据——产出物本身就是验证手段,不是验证完成后的附加文档。

**环境隔离建议**:如果可能,让每个并行的任务/worktree 对应一份独立可启动的应用实例,这样多个 agent 同时验证不同改动时不会互相污染彼此看到的状态。

### 2. 可观测性回路

适用于:性能预算、可靠性约束、跨服务的行为验证("这四条关键用户旅程里,任何一个 span 都不能超过 2 秒")。

标准做法:

- 应用要输出**结构化**日志、指标、追踪(不是自由文本日志,否则没法被查询)。
- agent 通过查询接口(日志查询语言、指标查询语言、追踪查询语言——具体用什么取决于项目栈)去验证约束,而不是读代码猜测运行时行为。
- 把验证目标写成可查询的断言,例如:"服务启动完成时间 < 800ms"、"P99 延迟 < 2s"、"某个错误日志在过去 N 次运行里出现次数为 0"。
- 同样建议为每个隔离的工作环境配一份临时的可观测性数据(日志/指标/追踪），任务结束后随环境一起清理，避免历史数据互相干扰判断。

## 把两者串起来用

完整的验证回路通常是:实现修复 → 用可观测性数据确认底层行为符合预期 → 用浏览器驱动确认用户可见的结果符合预期 → 把这两类证据都附在 PR/exec-plan 里,而不是只写"应该修好了"。

## 操作步骤

1. 明确这次要验证的是"用户可见的行为"还是"底层运行时约束",或者两者都要。
2. 如果是 UI 相关:执行浏览器驱动验证循环,产出修复前后对比证据。
3. 如果是性能/可靠性相关:把约束写成可查询的断言,用可观测性工具验证。
4. 把验证结果(截图、查询结果)附在改动说明里,作为"完成定义"的一部分——所有自动化检查必须通过、没有未处理的评审意见,才算验证完成。
5. 如果发现项目缺少必要的观测能力(没有结构化日志、没有浏览器自动化接入),把这个缺口本身记录下来——按 `harness-verification-loop` 的思路,这是一个该被修的"环境缺失",不是这次任务的失败。

## 配合的 agent

- `qa-verifier` agent:专门执行这个技能描述的验证循环,产出证据,本身不负责修代码。
